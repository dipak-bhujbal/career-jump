# Release Notes — v2.1.0

**Release date:** 2026-04-28  
**Branch:** main  
**Previous version:** v2.0.0

---

## Summary

This release delivers the Dynamic Plan Config system, making all pricing tier policy admin-configurable at runtime without code changes. It also introduces Stripe-powered upgrade flows, per-plan usage limits on jobs and applications, and a 4-tier pricing structure.

---

## What's New

### Dynamic Plan Config (admin-controlled pricing policy)

- **`PlanConfig` stored in DynamoDB** (`cj-billing` table, `pk=PLAN_CONFIG`). All plan limits are now live-editable from the admin portal — no redeployment required.
- **4 tiers:** `free`, `starter`, `pro`, `power` — with admin-editable display names, so "Pro" can be renamed to "Growth" without touching code.
- **Admin API routes:**
  - `GET /api/admin/plan-config` — returns all 4 plan configs
  - `PUT /api/admin/plan-config/:plan` — updates a single plan's settings
- **5-minute in-memory cache** per Lambda instance; `savePlanConfig` invalidates immediately.
- **Per-plan limits now configurable:**
  - `scanCacheAgeHours` — cache TTL for user-triggered scans
  - `canTriggerLiveScan` — false locks tier to cache-only (free)
  - `maxCompanies` — max tracked companies (null = unlimited)
  - `maxSessions` — concurrent session cap
  - `maxVisibleJobs` — available jobs page display cap
  - `maxAppliedJobs` — applied jobs tracking cap
  - `emailNotificationsEnabled`, `weeklyDigestEnabled`, `maxEmailsPerWeek`
  - `enabledFeatures` — feature gate list

**Default seeds:**

| Plan    | Cache  | Live scan | Companies | Sessions | Visible jobs | Applied jobs |
|---------|--------|-----------|-----------|----------|--------------|--------------|
| free    | —      | No        | 5         | 1        | 15           | 50           |
| starter | 4h     | Yes       | 10        | 1        | 40           | 150          |
| pro     | 8h     | Yes       | 25        | 2        | 100          | 500          |
| power   | 4h     | Yes       | unlimited | 3        | unlimited    | unlimited    |

### Usage Limit Enforcement

- **Available jobs cap:** `GET /api/jobs` truncates results to `maxVisibleJobs` and returns `totals.jobsCapped`, `totals.jobCapLimit`, and `totals.totalAvailableJobs` so the frontend can show "47 more jobs — upgrade."
- **Applied jobs cap:** `POST /api/jobs/apply` returns `402 { error: "applied_jobs_limit_reached", limit: N, plan }` when the user's tracked applications reach their plan cap. Existing applications are never deleted.
- **Session limit:** enforced dynamically from plan config (replaces hardcoded 1/2/3 values).

### Stripe Billing Integration

- **`stripe@17.7.0`** added to backend dependencies.
- **Stripe config stored in DynamoDB** (`pk=STRIPE_CONFIG`). Admin configures publishable key, secret key, webhook secret, and per-plan price IDs from the admin portal. Secret key is write-only — never returned by the API.
- **Admin routes:**
  - `GET /api/admin/stripe-config` — returns public config (no secrets)
  - `PUT /api/admin/stripe-config` — saves full config including secrets
- **User-facing routes:**
  - `POST /api/billing/checkout` — creates Stripe Checkout session, returns redirect URL
  - `GET /api/billing/subscription` — returns current subscription + plan config for profile page
  - `POST /api/stripe/webhook` — handles `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`; updates `BillingSubscriptionRecord` in DynamoDB automatically

### Frontend Upgrade Flows (codex)

- **Admin > Stripe Config** — new admin card to configure Stripe keys and price IDs per tier
- **Admin > Plan Config** — new admin page to edit all 4 plan tiers with live limit controls
- **Profile > Subscriptions tab** — current plan, billing period, usage summary, Upgrade/Manage CTA
- **Jobs page cap banner** — shown when `totals.jobsCapped === true`; "X more jobs hidden — upgrade to Pro"
- **Apply limit modal** — shown on `402 applied_jobs_limit_reached`; upgrade CTA redirects to Stripe Checkout
- **Post-login upgrade nudge** — one-shot banner for free/starter users after login

### Admin UI

- **Tenant ID** now shown as a copyable field in the Admin > Users detail panel

## Deployment Notes

### Required environment variables
No new env vars required — Stripe keys are stored in DynamoDB via admin UI, not in environment variables.

### DynamoDB
No schema migrations needed. New items written to existing `cj-billing` table:
- `pk=PLAN_CONFIG, sk=free|starter|pro|power` — written on first admin save (defaults served from code until then)
- `pk=STRIPE_CONFIG, sk=CONFIG` — written when admin configures Stripe

### Post-deploy checklist
1. Deploy backend (`sam deploy --config-env prod`)
2. Verify CloudWatch logs — scheduler, scanner, ops dashboard Lambda
3. Open Admin > Plan Config — confirm all 4 tiers load with default values
4. Open Admin > Stripe Config — enter Stripe keys and price IDs for each paid tier
5. Configure Stripe webhook endpoint: `https://api.careerjump.io/api/stripe/webhook`
   - Events: `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`
6. Test checkout flow end-to-end with a Stripe test key before switching to live keys
7. Confirm `GET /api/billing/subscription` returns correct plan for a test user

---

## Breaking Changes

None. All existing plan behavior is preserved — hardcoded defaults match previous constants exactly. Admin can adjust limits post-deploy without redeployment.
