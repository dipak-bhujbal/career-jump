# Career Jump AWS v3.1.0

**Released:** 2026-04-23

## Summary

Fixes two root causes that allowed user-discarded jobs (e.g. Nvidia) to silently reappear in the Available Jobs list after subsequent scans.

## Bug Fixes

### 1. Discard key pruning bug (Critical)

**Symptom:** Jobs discarded on Day 1 reappear after the user discards a *second* job and runs another scan.

**Root cause:** `pruneDiscardedJobKeysToKnownJobs` ran every time `saveInventory` was called ŌĆö including from the discard route. The prune function builds `knownKeys` from the current stored inventory, which never contains previously-discarded jobs (they are intentionally absent). So older discard keys (e.g. Nvidia) were silently removed from the registry whenever a new discard triggered a `saveInventory` call.

**Fix:** `saveInventory` now accepts `options?: { skipKeyPrune?: boolean }`. The discard route passes `skipKeyPrune: true`. Key pruning only runs during full scan saves (`runScan`, `finalize-run`), when the complete raw inventory (including ATS-fetched jobs for all companies) is in memory and `knownKeys` is comprehensive.

### 2. ATS job ID churn (Major)

**Symptom:** A discarded job reappears after a scan even though the discard key was intact ŌĆö because the ATS rotated the numeric job ID in the URL between runs.

**Root cause:** The discard registry stored only `jobKey` (`source:company:id`), which is ATS-volatile. A new ID means a new key ŌĆö the old discard entry never matches.

**Fix:** The discard registry is upgraded to `{ jobKey: string, fingerprint: string }` pairs using the existing `jobStableFingerprint` (title + URL-token hash). Discard suppression now matches on **either** the exact row key **or** the stable role fingerprint.

## Architecture Changes

### Discard Registry

The `DISCARDED_JOB_KEYS_KEY` DynamoDB entry upgrades from `string[]` to `Array<string | { jobKey: string; fingerprint: string }>`. Plain-string legacy entries remain valid (read as `jobKey` only ŌĆö no migration needed).

### New types and helpers

- `DiscardRegistry: { jobKeys: Set<string>; fingerprints: Set<string> }` ŌĆö the loaded registry
- `loadDiscardRegistry(env, tenantId)` ŌĆö replaces internal `loadDiscardedJobKeys`
- `isDiscarded(job, registry)` ŌĆö single predicate used by `buildInventory`, `pruneInventoryForStorage`, and `getLatestRunNotificationJobs`

### Pruning policy (asymmetric by design)

| Dimension | Prune policy |
|---|---|
| `jobKeys` (legacy string entries) | Pruned at scan time if key is no longer in full scan inventory or applied jobs |
| `fingerprints` (structured entries) | Never auto-pruned ŌĆö durable across ATS id rotation |

Fingerprints accumulate at most one entry per job ever discarded. Storage cost is negligible. TTL-based expiry can be added in a future release if needed.
