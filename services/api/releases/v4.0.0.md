# Career Jump AWS v4.0.0

**Released:** 2026-04-25

## Summary

Major architectural release. The ATS layer is rearchitected around a registry-driven dispatcher with a pluggable adapter interface, the per-company job processing path is replaced by a composable pipeline (filters / enrichers / reducers), and Phase-2 multi-tenant scaffolding (DynamoDB key conventions + tenant settings type) is laid down without changing single-tenant behavior.

This is a foundation release for the SaaS path — it does not change end-user behavior on its own, but it unblocks per-tenant data isolation, registry-driven scraping, and richer job filtering/normalization in subsequent releases.

## Highlights

- **15 ATS adapters** in `src/ats/core/` (was 5): `ashby`, `bamboohr`, `breezy`, `eightfold`, `greenhouse`, `icims`, `jobvite`, `lever`, `oracle`, `phenom`, `recruitee`, `smartrecruiters`, `successfactors`, `taleo`, `workable`, `workday`.
- **Custom adapters** in `src/ats/custom/` for sites that don't fit a known ATS: `apple`, `berkshire`, `tesla`, plus generic `jsonld` (schema.org JobPosting extraction) and `sitemap` fallbacks.
- **Registry-driven dispatch** — a single seed registry (`data/seed_registry.json`) maps company → ATS → adapter; a unified `AtsAdapter` interface and self-registration pattern replaces ad-hoc per-adapter wiring.
- **Composable jobs pipeline** (`src/jobs/`) with stages for country / keyword / job-title / location / posted-date / department filtering, location / seniority / comp-type / salary / fingerprint enrichment, and dedupe / relevance-rank reduction.
- **Multi-tenant scaffolding** — `tenant#<id>` DynamoDB key convention, optional `RuntimeConfig.tenantId`, and `TenantSettings` type. Default tenant is `"default"`; current single-tenant deploys are unaffected.
- **Registry API** — five new `/api/registry/*` endpoints for inspecting and on-demand scraping.
- **Test infrastructure** — vitest, fixtures, schema-check tooling.

## Architecture Changes

### ATS layer reorg

- Existing adapters moved from `src/ats/*.ts` → `src/ats/core/*.ts`.
- New `src/ats/index.ts` barrel re-exports legacy paths so existing call sites continue to work.
- New `src/ats/registry.ts` loads the seed registry and dispatches to the right adapter via the shared interface.
- New `src/ats/shared/` provides:
  - `http.ts` — central `atsFetch` with consistent UA, default 15s timeout, retry with exponential backoff on 408/429/5xx.
  - `init-core.ts` — registers all core adapters at module load.
  - `normalize.ts` — `normalizeAtsId` to canonicalize variants ("greenhouse-embed", "wd1.workday.com" → `greenhouse`, `workday`).
  - `slug.ts` — shared URL-derivation helpers used by multiple adapters.
  - `types.ts` — `AtsAdapter` interface, adapter registry, `AdapterConfig`, `FetchOptions`.
- New `src/ats/custom/` — non-ATS scrapers (Apple Jobs, Berkshire careers index, Tesla, generic JSON-LD, generic sitemap). Self-registers via `custom/index.ts`.

### Jobs pipeline (`src/jobs/`)

A staged async pipeline replaces ad-hoc transformations:

```ts
const out = await pipe(rawJobs, ctx,
  enrichers.normalizeLocation,
  filters.byCountry(["US"]),
  enrichers.extractSeniority,
  enrichers.extractCompType,
  enrichers.extractSalary,
  enrichers.computeFingerprint,
  reducers.dedupeByFingerprint,
  reducers.rankByRelevance,
);
```

Stages are typed (`Filter | Enricher | Reducer`) and run sequentially — order matters (enrichers before filters that depend on enriched fields). `runStateful` and `buildPipeline` are available for stages that carry state.

### Multi-tenant scaffolding (Phase 2 prep)

- `src/storage/tenant-keys.ts` — DynamoDB key helpers. Pattern: `pk = "tenant#<id>#<entity>"`. `DEFAULT_TENANT = "default"` keeps single-tenant deploys working without migration.
- `RuntimeConfig.tenantId?: string` — optional. When unset, behavior is single-tenant.
- `TenantSettings` type — per-tenant `displayName`, `jobtitlesOverride`, `enabledRegistrySheets`, `pipelineOverrides`. Stored at `pk = "tenant#<id>"`, `sk = "settings"`.
- `src/storage/registry-cache.ts` — Lambda-resident registry cache, designed so the JSON-file source can be swapped to a per-tenant DynamoDB read without changing call sites.

### Shared HTTP

All new adapters route through `atsFetch` for:
- Consistent `User-Agent: career-jump/1.0`
- Default 15s timeout (slow ATS can't stall the pipeline)
- Retry on transient 408/429/500/502/503/504 with exponential backoff (default 2 retries, 500ms base)
- `AbortController` integration for nested signals

## New API Endpoints

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/registry/meta` | Registry metadata + adapter list + tier counts |
| GET | `/api/registry/companies?ats=X&tier=Y&limit=N` | List registry companies, optionally filtered |
| GET | `/api/registry/companies/:name` | Get one registry entry by company name |
| POST | `/api/registry/scrape` | On-demand scrape one company; body: `{ company, applyPipeline? }` |
| POST | `/api/registry/scrape/ats` | Batch scrape all companies for an ATS; body: `{ ats, applyPipeline?, limit? }` |

These endpoints route to `services/registry-scraper.ts` and apply the default jobs pipeline when `applyPipeline=true`.

## Backwards Compatibility

- **Adapter call sites unchanged.** `src/ats/index.ts` re-exports the legacy `fetchAshbyJobs`, `fetchGreenhouseJobs`, etc., from their new `core/*` locations. Existing imports continue to work.
- **Single-tenant behavior unchanged.** `tenantId` is optional throughout; absence means default tenant. No DynamoDB migration required.
- **Discard registry, scan flow, public API for jobs / applied-jobs / dashboard / action-plan** — untouched.
- **Frontend** ([public/app.js](public/app.js), [public/index.html](public/index.html), [public/logs.html](public/logs.html), [public/styles.css](public/styles.css)) is not modified in this release. Version labels in HTML still read 3.1.0; they will be updated when a user-visible feature lands on top of this foundation.

## Deployment Notes

- All four Lambda functions (ApiFunction, ScanCompanyFunction, FinalizeRunFunction, RunOrchestratorFunction) need to be redeployed with the new bundle.
- No DynamoDB schema migration required.
- New `data/seed_registry.json` must ship in the Lambda bundle — verify the SAM build includes it (the registry-cache loader reads from `data/seed_registry.json` relative to the deployed module root).
- Frontend S3 sync not required for this release.

## Rollback

- The release branch (`release-4.0`) and tag (`v4.0.0`) can be deleted on the remote.
- Lambda rollback: redeploy from the previous tag (`v3.1.0`).
- No data shape changes — rollback is fully backwards-safe.
